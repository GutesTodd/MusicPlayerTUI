# КОНТЕКСТ ПРОЕКТА
Я разрабатываю TUI-клиент для Яндекс Музыки (`yandex-music-cli`) на базе фреймворка Textual.
Архитектура разделена на слои:
- `shared/domain` — сущности и интерфейсы.
- `backend/contexts/...` — бизнес-логика (Use Cases) и роутеры (взаимодействие через Unix Sockets).
- `backend/infrastructure` — реализации провайдеров (Yandex Music API, MPV Player и т.д.).
- `ui/` — клиентская часть на Textual.

# ЗАДАЧА
Мне нужно унифицировать логику создания очереди воспроизведения (`TrackQueue`). Сейчас трек может прийти из разных мест: как одиночный трек из поиска, как часть альбома, как часть плейлиста или как топ-треки артиста.

## Требования к реализации:
1. **Слой Use Cases (`backend/contexts/playback/use_cases/`):**
   - Спроектируй и реализуй Use Case (например, `PlayMediaUseCase` или `EnqueueMediaUseCase`), который будет выступать фасадом.
   - Он должен принимать некий универсальный идентификатор (например, ID и тип медиа: track, album, playlist, artist).
   - Под капотом Use Case должен использовать правильный провайдер (`TrackProvider` или `TrackSearcher`), получать нужную сущность, прогонять её через `QueueFactory.create_queue` и обновлять состояние в `QueueManager`.
   - Напиши тесты для этого Use Case (используй TDD, как требует фреймворк).

2. **Слой UI (`ui/views/search_view.py` и `ui/viewmodels/player.py`):**
   - Изучи документацию Textual (если нужно). 
   - Обнови UI так, чтобы при выборе элемента в результатах поиска (одиночного трека или альбома) генерировалось событие, которое через сокет вызывает наш новый Use Case на бэкенде.
   - Код Textual должен оставаться реактивным (используй Messages/Events или реактивные свойства viewmodel).

## Порядок действий (для фреймворка Superpowers):

2. Напиши мне короткий Design Document: как ты предлагаешь связать UI-событие выбора медиа с формированием очереди в `QueueManager`. Какие DTO мы будем передавать между клиентом (UI) и бэкендом? Жди моего подтверждения (или задай наводящие вопросы), прежде чем писать код.
